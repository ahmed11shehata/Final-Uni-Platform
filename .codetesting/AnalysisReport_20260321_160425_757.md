# Test Failures due possible code bugs

## AYA_UIS.Application.UnitTests.csproj - AYA_UIS.Application.Handlers.Assignments.UnitTests.GetAssignmentSubmissionsQueryHandlerTests.Handle_MapsSubmissionPropertiesAndResolvesStudentNameAndFeedback
- **Confidence**: High
- **Test File**: AYA_UIS_Server\AYA_UIS.Application.UnitTests\Handlers\Assignments\GetAssignmentSubmissionsQueryHandlerTests.cs
- **Bug Location**: AYA_UIS_Server\AYA_UIS.Application\Handlers\Assignments\GetAssignmentSubmissionsQueryHandler.cs@29

### Analysis
The failure is caused by the production mapping logic in GetAssignmentSubmissionsQueryHandler. The handler sets StudentName using null-coalescing (s.Student?.DisplayName ?? s.Student?.UserName ?? "Unknown"). This only falls back when a value is null, but not when DisplayName is an empty string. In the test submissionWithUserNameOnly the Student.DisplayName is string.Empty (not null), so the expression returns empty string and the fallback to UserName doesn't occur. The test expectation (use UserName when DisplayName is empty) is reasonable. The fix is to treat empty or whitespace DisplayName (and UserName) as missing by using string.IsNullOrWhiteSpace checks rather than only null checks.

### Suggested Fix
Replace the null-only coalescing logic for StudentName with explicit empty/whitespace checks. For example, change the StudentName assignment in GetAssignmentSubmissionsQueryHandler.Handle to:

```csharp
StudentName = !string.IsNullOrWhiteSpace(s.Student?.DisplayName)
    ? s.Student!.DisplayName
    : !string.IsNullOrWhiteSpace(s.Student?.UserName)
        ? s.Student!.UserName
        : "Unknown",
```

This ensures that an empty DisplayName will not be used and the code will fall back to UserName, then to "Unknown" if both are null/empty. The null-forgiving operator (!) is safe here because the preceding null-or-whitespace checks handle the null cases. Alternatively, you may write a small helper method to encapsulate the fallback behavior if preferred.

## AYA_UIS.Application.UnitTests.csproj - AYA_UIS.Application.Mapping.UnitTests.MappingProfileTests.MappingProfile_Constructor_ConfigurationIsValid
- **Confidence**: High
- **Test File**: AYA_UIS_Server\AYA_UIS.Application.UnitTests\Mapping\MappingProfileTests.cs
- **Bug Location**: AYA_UIS_Server\AYA_UIS.Application\Mapping\MappingProfile.cs@32

### Analysis
The test constructs the MappingProfile and builds an AutoMapper configuration, then calls AssertConfigurationIsValid(). AutoMapper throws DuplicateTypeMapConfigurationException because MappingProfile declares the same mapping pair (Fee -> FeeDto) twice. In MappingProfile.cs there is an earlier CreateMap<Fee, FeeDto>().ReverseMap(); and later another CreateMap<Fee, FeeDto>() with a ForMember(...) and ReverseMap(). AutoMapper requires a single CreateMap per source/destination pair. The test is correct to validate the profile; the production mapping profile must be fixed to define the Fee -> FeeDto mapping only once (merging the configuration options into that single mapping).

### Suggested Fix
Remove the duplicate CreateMap declaration for Fee -> FeeDto and keep a single mapping that includes the DepartmentName ForMember mapping and ReverseMap(). In other words, delete (or merge) the earlier CreateMap<Fee, FeeDto>().ReverseMap(); so there is only one CreateMap<Fee, FeeDto>(...). ReverseMap() should be kept on that single mapping. This eliminates the DuplicateTypeMapConfigurationException.
